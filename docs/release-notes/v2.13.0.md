# jawata v2.13.0 — the profiler

**Sprint 24, phase 3.** jawata can already watch a program run and stop it where it goes
wrong. Now it can tell you *why it is slow*, *what it is holding*, and *what killed it*.

One new front door — `profile` — process-level diagnostics against any session `debug`
already opened. It needs no debug agent to be useful: threads, heap, GC and native memory
are facts about the *process*, not facts behind a debug channel.

**45 tools** (was 44).

---

## What it does

**The profiling floor.** `threads` (every OS thread, with its Java stack, even mid-breakpoint
— it reads the process, not the suspended JDI view), `deadlock` (from the *same* dump
`threads` reads, so the two can never disagree — a wait chain is *named*, not just dumped),
`histogram`, `gc`, `nmt`, `heap_dump`. Every answer is parsed and structured; a capability
the target genuinely lacks — native memory tracking that was never turned on — says so
plainly, `enabled: false` and why, never a fabricated all-zero report.

**Profiles that name symbols.** `sample` and `jfr_dump` capture Java Flight Recorder data —
targeted or on-demand, mid-run, no new sampling window needed. `hotspots` ranks it by
CPU, allocation, lock contention, or GC, and the top row **is a symbol**:
`ClassName#methodName`, the exact key `get_call_hierarchy` and `find_references` already
take. Not a line in a flame graph you have to go translate — the same name, everywhere.

**Latency at a seam you name.** `latency_seam` traces a method non-invasively (the same
mechanism `debug`'s probes use — the program is never stopped) and reports p50/p99/p999
*twice*: raw, and corrected for coordinated omission. Most latency traces are closed-loop —
the next call waits for the last one to finish — which silently hides exactly the stalls a
latency measurement exists to catch. Both numbers are reported so the correction's effect is
visible, never asserted.

**One alarm, one bundle.** `incident_arm` watches a file path the *target* writes to when it
decides something is wrong; the first poll after that file appears captures a seven-part
bundle — JFR slice, thread dump, heap histogram, heap dump, log slice, replay descriptor, and
a summary that **names the alarming symbol**, read from the target's own declaration. jawata
orchestrates and captures; the target says what actually broke.

**Read what the JVM already exposes.** `jmx_read` and `log_level` (with an optional
automatic revert on expiry, so a diagnostic verbosity bump cannot outlive the reason you set
it) — over the same local JMX connection the dev/sim preset already enables.

**When the crash is native.** `native_hs_err` parses the JVM's own crash report — and the
finding underneath it is the interesting part: **HotSpot already correlates Java and native
frames for the crashing thread, for free.** No external debugger is needed to see that a
`SIGSEGV` inside `Unsafe_PutInt` traces back to your own method three frames up — the JDK's
own fatal-error report already says so. `native_nmt` reads the memory-tracking section
embedded in that same report (only present if it was turned on before the crash — nothing
can be enabled after the fact). `native_handoff` is the honest *optional* step beyond that
baseline: hand the crash to a configured gdb or lldb for a deeper backtrace, correlated
against what hs_err already resolved — and where no adapter is configured or installed, it
says so, `available: false` and why, never a silent skip.

## The loop closes, on the profiler too

A hotspot ranking already carries a compiler-accurate symbol. That symbol goes **straight**
into `get_call_hierarchy` — the same claim v2.12.0 proved for a breakpoint hit, now proved
for a measurement. A search would only re-derive what the profiler just measured, and could
rank something else entirely.

---

## Read this before you point it at something

The same three statements v2.12.0 made about `debug` hold for `profile`: it is meant for a
development or simulation machine; a production JVM you did not choose to expose this way is
not merely refused, it is unreachable by construction; where a JVM *is* reachable, that is
your professional judgment, and jawata will not second-guess it.

---

## Upgrading

Drop-in. No configuration changes, no new dependencies — Java Flight Recorder and jcmd are
already part of every JDK 21 that runs jawata; the gdb/lldb adapter is opt-in and entirely
optional. Nothing you were doing behaves differently; there is simply more `debug` can tell
you about the same session.
