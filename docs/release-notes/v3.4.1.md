# jawata-mcp v3.4.1 — the recall v3.4.0 promised, actually running

v3.4.0 shipped its central feature switched off. The embedding model loaded, the
vectors were written, and nothing ever consulted them: every place in the running
product that answers a question built its recall without the meaning-based half.
Asking in different words got you the same keyword search as v3.3.1.

It was caught by using the product, on the first real question, hours after
release. This patch fixes it and three further defects found alongside.

## What changes for you

**Asking in your own words now works.** A note recorded as "the roof leaked
because nobody swept the gutters" is found by asking about "water through the
ceiling after the drains clogged" — no shared words. That is what the previous
release described and did not do.

**Your existing notes become findable.** Entries written before this version had
no vectors and were invisible to meaning-based search. Each start now embeds a
batch of them in the background, logging how many remain. A large store becomes
fully searchable over a few restarts, and answers by keyword throughout.

**Notes you rejected stay gone.** Rejected and superseded entries were excluded
from keyword results but not from meaning-based ones — invisible while nothing
consulted that path, and live the moment it was connected. Refused knowledge
never comes back now, by any route.

**The quality counters work on an existing store.** Their table was created only
for stores upgrading *to* the previous version, so a store already upgraded never
received it and recorded nothing. It now gets its own upgrade step.

## Unchanged

Everything else in v3.4.0 stands: facts stay address-bound and terminal,
experience arrives as capped labeled analogy, no similarity number appears in any
answer, the precedent warning charges only on an exact repeat, and a broken
embedder still degrades to v3.3.1 keyword behaviour. Tool count remains 45.

A question with no answer still answers "nothing" — asking about something the
store has never seen returns an honest absence, not the nearest thing available.

## How this was verified

A new end-to-end test (`build/end-to-end-test.sh`) starts the built product as a
separate process and asks it, over the same connection an editor uses, to perform
each promise. It has no way to reach inside and supply what is missing — which is
exactly what every unit test was unknowingly doing.

Run against the previous release it fails, naming the broken promise. Run against
this one it passes. The 1592 automated tests also pass, though it is worth saying
plainly that they passed on the broken release too.
