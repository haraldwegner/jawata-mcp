# JAWATA v2.7.1 — dogfood hardening

Two defects found the day v2.7.0 shipped — by pointing jawata at its own codebase —
fixed the same day.

## Fixed

- **`find_duplicate_code` could drop the connection.** The clone-scan tokenizer's
  regex burned one backtracking stack frame per character of a string literal;
  a literal past ~2k characters threw `StackOverflowError`, which no
  `catch (Exception)` sees — the request thread died and the client saw a closed
  socket instead of an error. The tokenizer now pre-collapses literals and comments
  in a linear scan (no recursion, any literal size), and it lexes **text blocks**
  correctly, which the old regex mis-read.
- **No tool call can kill the connection anymore.** The tool dispatcher now converts
  *any* JVM `Error` into a structured `INTERNAL_ERROR` response. A failed request
  answers; it never disconnects.
- **`compile_workspace` output is bounded.** A classpath-broken workspace returned
  17,383 diagnostics as one 4.25 MB response. The tool now pages like
  `find_quality_issue` (v2.6.1): `limit` (default 100) / `offset`, plus
  `summary:true` for counts-only with a per-project breakdown.
  `errorCount`/`warningCount`/`count` always reflect the full set.

## Compatibility

- The v2.7.0 legacy fallbacks (`goja.*` system properties, `GOJA_*` environment
  variables) **remain in place for this patch release**; their removal moves to the
  next minor release. Switch to the `jawata` names now.
