# jawata v2.11.1 — dogfood patch on v2.11.0

v2.11.0 shipped memory-first navigation this morning. Working it in anger on
jawata's own workspace for ten minutes found two things 1249 green tests could
not — because both are failures the *fixtures were too small to expose*.

That is the case for dogfooding every release rather than hoping.

## Landmarks: a ranking that could not rank

The reference count behind `inspect(kind=landmarks)` was capped at 200 to keep
the scan cheap. On a real 506-source workspace, **the top six types all hit that
cap at once** — `IJdtService`, `JdtServiceImpl`, `ToolResponse`, `ResponseMeta`,
`RefactoringChangeCache`, `TestProjectHelper`, every one reporting exactly
`200`.

So the ordering among precisely the types the feature exists to rank was
arbitrary, and the number looked like a count when it was a floor. The one
question landmarks answers — *what is this codebase built on?* — was the one it
could not answer.

- The bound is now high enough to discriminate (2000).
- A count that still hits the bound is reported as what it is: a floor, flagged
  `"atLeast": true`. Never a floor dressed as a count.

## Relocate: three different truths, no longer told as one

`SYMBOL_RELOCATED` was answering three quite different situations with one
shape, and getting two of them wrong.

**A typo now gets a confident correction.** Asking for
`IJdtService#getLineNumberr` correctly leads with `getLineNumber` — but it used
to drag `getPathUtils` and `getProjectRoot` along too, members sharing nothing
with the query but a `get` prefix. Suggestions are now filtered by name
affinity: a wrong suggestion is worse than one fewer suggestion.

**A member renamed to an unrelated word gets honesty instead of a guess.**
`multiply` → `times` share not one letter; nothing in the index says the one
*became* the other. Answering "Found: times" would have been a guess wearing a
fact's clothes. It now says the member is not there and names the ones that
are — a directory, for you to pick from, not a claim.

**A member missing from a type that still exists no longer claims the type is
gone.** Previously `Calculator#somethingWrong` could answer *"it is gone, not
moved"* while `Calculator` sat right there. It now says what is actually true:
the type did not move; that member is not on it — and here is what it has.

## Also

- The new name form (`symbol=` / `typeName=`) turns out to work through
  existing MCP client sessions without a restart — better than the schema-cache
  caveat in the v2.11.0 notes predicted.

## Tests

Suite 1251 = 1251 passed, 0 skipped (was 1249). All three dogfood findings are now
locked in by tests, so they cannot come back.

## Never claim an absence you cannot prove

A third finding, this one surfaced by the test suite under load rather than by
live use. Right after a refactoring, while the Java model is still settling, a
lookup for a type that plainly exists can come back empty. `SYMBOL_RELOCATED`
then inferred *"it is gone, not moved"* — asserting an absence from a lookup
that had merely failed, about a type sitting right there in the workspace.

A failed lookup proves only that the lookup failed. Two changes:

- The type is now resolved **once**, and that single fact drives every branch —
  two independent resolutions could disagree with each other.
- A member-form name whose type will not resolve now says only what was
  observed ("no type `X` was found"), never that it is gone. The strong "gone,
  not moved" claim is reserved for the case where a positive search also came
  back empty.

It would have been easy to write this off as a flaky test — it passed 5/5
focused and clean serially, and its two neighbours in that run genuinely *were*
contention noise. It was not noise; it was the code stating something false.
