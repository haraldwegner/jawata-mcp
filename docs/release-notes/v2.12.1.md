# jawata v2.12.1 — refactorings that check their own work

**Sprint 24, Stage 14 — the dogfood patch on v2.12.0**, from the first live
self-refactoring session on jawata's own codebase, hours after the release.

## Every applied refactoring now verifies its own result

When a refactoring tool applies a change, it immediately compiles the files it
modified. If the change introduced compiler errors, **the tool undoes it and
refuses — with the compiler's own messages** — instead of answering
`applied: true` and leaving the breakage for you to discover. Successful
responses carry `compileVerified: true`, and that word now means something: it
was checked, not hoped.

One deliberate exception, stated in the response rather than hidden:
`change_method_signature` legitimately leaves bodies and call sites to adapt
(changing a return type makes old `return` statements red *by design*). There,
type errors are kept and **listed in the response** (`compileVerified: false`
+ `introducedErrors`) so you fix them immediately; syntax errors are never
legitimate output and are still undone and refused.

## `extract_method` runs on Eclipse's own engine

The extraction is now computed by JDT's `ExtractMethodRefactoring` — the exact
engine behind the IDE's Refactor → Extract Method, with its exhaustive dataflow
analysis. Selections it cannot transform are **refused with its reasons**
(`EXTRACT_REFUSED`) instead of guessed at. The previous implementation
mis-handled a variable declared inside the selection and used after it; that
shape now round-trips correctly — and is pinned by a dedicated test.

## Renaming a type renames its file

`rename_symbol` on a class whose file bears its name now renames the file in
the same change (`fileRenamed` in the response), undo restores both. Previously
the file kept the old name and the response carried a note advising you to
notice — a state the new verify gate would rightly have refused.

## Also fixed

- **`extract_interface` generates an interface that compiles.** The generated
  file carried no imports, so any signature using an imported type
  (`List<String> getItems()`) produced an interface the class could not
  actually implement. The source file's imports now travel with the extracted
  signatures. (Caught by the new verify gate on first contact.)
- **`change_method_signature` placeholder arguments compile.** A freshly added
  parameter at a call site used to be filled with a bare `/* TODO */` comment —
  a syntax error by construction. It is now a typed zero value with the TODO
  beside it: `/* TODO count */ 0`.
- **`debug(action=breakpoint_set)` validates kind-specific parameters.** A
  method breakpoint without `method` (likewise field watch without `field`,
  conditional without `condition`, hit-count without `hitCount`) is refused at
  set time — it used to be accepted as a successful "pending" breakpoint that
  could never bind, with a misleading "class not loaded yet" reason.
- **Internal errors name the exception.** `Internal error: null` (a bare NPE
  message) is now `NullPointerException at <frame>` — a failure you can act on.

## Upgrading

Drop-in. Existing behavior changes only where the old behavior produced or
tolerated broken code.
