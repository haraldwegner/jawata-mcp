# jawata v2.13.1 — the profiler, corrected

**Sprint 24 patch.** After v2.13.0 shipped, we put it through a hard, fresh-eyes audit —
the spec read against the actual code, one reviewer per area, each told to assume a feature
had been quietly narrowed until the code proved otherwise. It found real problems: answers
that were confidently wrong, features the spec asked for that had been left out, and a few
places that could hang or leak. This release fixes **all** of them.

**Still 45 tools.** Drop-in. Nothing you were doing behaves differently — several things you
were doing now behave *correctly*.

---

## Wrong answers, now right

- **`profile(threads)` reported the wrong thread id for every thread** — a number-base
  mistake turned each id into a different, nonexistent one, so cross-referencing `top -H` or
  `kill -QUIT` landed nowhere. Fixed; the id is the JVM's own.
- **Native crash correlation never actually fired.** `native_handoff` compared a bare symbol
  (`Unsafe_PutInt`) against the crash report's `Unsafe_PutInt+0xa4` and never matched, so the
  gdb/lldb adapter's whole value-add was dead code. Symbol names are now normalized before
  matching (offset, C++ signature, `[clone]` marker stripped) and the correlation fires.
- **An incident that alarmed with plain text wedged itself** — the fallback meant to tolerate
  a non-JSON alarm threw instead, and the capture window (a live process at the moment it went
  wrong) was lost. Fixed.
- **`jcmd` / gdb could hang forever.** A wedged target held the call open with no timeout;
  now it drains on its own thread and fails at the deadline as documented.
- **A chatty program launched under the debugger could deadlock on its own output.** Its
  stdout is now drained for the target's life, so a replay that logs per event no longer
  stalls before reaching what you came to see.
- **A costly conditional or logpoint could freeze the whole session** by evaluating on the
  event-pump thread. Evaluation now runs off that thread with a timeout, and a condition that
  times out is dropped rather than wedging every later hit.
- **Leaks closed:** failed incident captures no longer strand invisible artifact directories
  (with multi-GB heap dumps inside); a stacked `log_level` bump reverts to the true baseline
  and cannot outlive a jawata restart; the landmark cache no longer serves a stale ranking
  forever on a long-lived host; expired artifacts are actually pruned.

## Missing pieces, now built

The spec asked for these; v2.13.0 shipped without them, and they were not recorded as skipped.
They are now built, not excused away:

- **Wall-time hotspots.** `hotspots(dimension=wall)` ranks by real elapsed time — on-CPU
  samples *plus* the time threads spend blocked on locks, I/O and parks — which is what you
  want when the slow thing is waiting, not computing. CPU, alloc, lock and gc were already there.
- **Real call counts.** `hotspots` call counts now come from an actual invocation counter, not
  a relabeled top-of-stack sample count (a method called 10,000× at a microsecond each used to
  read as ~zero).
- **Your program's own events.** `latency_seam` now surfaces the domain events the target
  itself emits, alongside the JVM's built-in ones.
- **Recall before you probe.** A debug session now recalls a previously recorded incident for
  the symbol you are about to investigate *before* you probe it — the memory arrives when it
  can still change what you do.
- **No recording left behind.** A JVM the debugger launches runs a continuous flight recording;
  its chunk directory is now owned by jawata and deleted on teardown, instead of leaking on
  every launch.
- **Capabilities read from the JVM, not guessed.** What a target can do
  (`nativeMemoryTracking`, `flightRecording`, `quietConsole`, `profilerReady`) is now discovered
  by asking the running JVM via jcmd — so a JVM prepared without our marker reports what it can
  actually do, instead of a fabricated `false`.

## Honesty note

The audit also found that our own close-out claimed "nothing was narrowed," which was not true.
That record — the dossier and the signed spec — has been corrected, and the full audit report
ships in the repository (`docs/sprints/audit-24-implementation.md`). The process lesson is
recorded too: a checkpoint has to test the whole requirement, not the one-line summary of it,
and a "nothing was left out" claim has to be checked by someone who did not write the code.

## Upgrading

Drop-in. No configuration changes, no new dependencies. If you were on v2.13.0, everything you
relied on still works — several things simply stopped lying.
