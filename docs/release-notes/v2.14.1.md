# jawata v2.14.1 — dogfood polish on v2.14.0

**Sprint 25 patch.** After v2.14.0 shipped, we drove the migrated refactorings and the
new smell kind in anger on jawata's own tree and put the release through a
fresh-context implementation audit (spec deliverable bodies vs the code, one auditor
per area, each assuming a feature had been narrowed until the code proved otherwise).
That surfaced a batch of small, real defects — several the audit found, a couple the
dogfood found, and two genuine bugs the audit's own new tests exposed. This release
fixes them.

**Still 45 tools.** Drop-in. Same names and parameters (plus a couple of new optional
ones).

---

## Wrong answers / broken behavior, now fixed

- **`generate` emitted TAB-indented code into space-indented files.** Every
  generated getter/setter/constructor/`toString`/`equals`/`hashCode`/override and
  `copy_class` came out with JDT's built-in tab default, regardless of the file's own
  style. Generated code now **matches the target file's indentation** by precedence:
  an explicit `indentChar` override, then the project's own formatter config
  (`.settings` — a repo that uses tabs still gets tabs), then a spaces/4 default for
  projects with no config.
- **`javadoc_lack` skipped code it shouldn't and missed code it should flag.** Two
  detector bugs: public methods like `issue()` / `settle(x)` were silently treated as
  trivial accessors (the `get`/`is`/`set` prefix check had no word boundary), and
  implicitly-public types nested inside interfaces/annotations were dropped entirely.
  Both fixed. It also now exempts **trivial constructors** (empty / single delegation
  or assignment) from the ratchet — the same rule it already applied to trivial
  accessors — so the documentation ratchet counts substance, not DI plumbing.
- **`search_symbols` result coordinates were documented inconsistently.** The output
  is 0-based line and column, matching every refactoring tool's 0-based input — feed a
  search result straight into a refactor, no arithmetic. The schema, description and
  steering now say so.
- **A relative `filePath` leaked an absolute mount path** into a tool response
  (`formatPath` resolved a relative input against the process working directory
  instead of the project root). Fixed; relative inputs come back project-relative.
- **`generate`'s `accessorKind` was unusable as documented** — the schema advertised
  `get`/`set`/`both` while the validator demanded `getters`/`setters`/`both`. The
  legacy values are now accepted and translated.

## New optional parameters

- **`rename_symbol` now accepts `symbol="pkg.Type#member"`** — the FQN form its
  sibling refactorings already had, and its steering already promised. It was declared
  in the schema but never wired; now it resolves to a position like `extract` and
  `find_references` do.
- **`generate(kind="getters_setters")` gains `indentChar`** (`space`|`tab`) — override
  the indentation of generated code for the rare case the project convention isn't
  what you want.
- **`extract(kind="constant")` gains `replaceAllOccurrences`** (default true, the IDE
  default) — replacing one of several identical literals and leaving the rest was the
  surprising outcome; opt out with `false`.

## Documentation and honesty

- `change_method_signature`'s retarget mode had a stale "no JDT equivalent" comment
  contradicting the corrected class Javadoc — removed.
- **Known bug, filed:** `organize_imports`' add-missing-import path throws deep in
  JDT's headless import rewrite (a null preference). The **prune-unused-static** half
  works (and is now tested); the add half is a ranked follow-up. Recorded in
  `parity/organize-imports/DIVERGENCES.md`.

## Note

The generated-code indentation default is tool-side and automatic — you don't pass it
per call. A studio-level default for projects with no formatter config is a planned
ergonomics setting (Sprint 28), not this release; correctness (matching the project)
ships here.
