# goja-mcp v2.4.0 — Strict disk sync: every answer reflects current disk (Sprint 21d)

goja now works on the current working tree the way an IDE does. External edits — the
AI agent's Edit tool, a `git checkout`, another editor — are visible to the very next
tool call: search, analysis, refactor change computation, and the knowledge store's
pointer judgments alike. No manual `refresh_workspace` needed, ever.

## How it works

A staleness guard runs once per tool call at the dispatch seam, before the tool
computes anything: a millisecond disk scan of every loaded project (mtime/size fast
path; content hash only for files inside the filesystem-timestamp granularity window,
where a same-size edit can hide), then `refreshLocal` on exactly the changed files and
one build per affected project — only when changes were detected. No file watcher, no
background threads, no race conditions: verification is synchronous inside the call,
and answers are as-of call start.

**There is no off switch.** Stale answers are a defect, and correctness is not
configurable — the only skip is the one the design earns per call: no edit detected,
no work. A guard failure is loudly logged (answers may be stale until the next
successful pass) and never blocks the call.

## Grounded judgments everywhere

- A file added externally — including a new file in a brand-new package — is found by
  the next `search_symbols`; a deleted one disappears; the diagnostics model walk
  includes it.
- A rename issued right after an external edit computes against the reconciled model —
  no stale-edit emission.
- The experience store's automatic `refresh()` judges symbol anchors against current
  disk: an entry re-anchored by an external rename is not mis-superseded, and stale
  anchors are correctly retired.

## Headless-JDT findings settled on evidence (in the code comments)

INCREMENTAL_BUILD's delta tracking lags fresh `refreshLocal` additions in this
headless runtime (search index follows, builder doesn't) → build is FULL, only on
change. Path→resource mapping resolves by location through the workspace root — goja
mounts sources as linked folders, so project-relative math is wrong by construction.
The Java model's package-fragment caches don't repopulate from resource deltas here —
structure changes close the source roots so the model relists lazily.

`refresh_workspace` remains available as the full clean-rebuild tool; its semantics
are unchanged. No tool-surface change; no schema change.
