# JAWATA v2.9.0 — the de-Tycho release (Sprint 22d)

jawata now builds with **plain Maven** and boots Eclipse's runtime itself. Same engine,
same 43 tools, same behaviour — proven by the full suite and live side-by-side parity —
minus the entire Tycho/p2 build system. For users nothing changes; for anyone building
from source, everything gets simpler and roughly five times faster.

## Changed

- **Build system: Eclipse Tycho → plain Maven.** Eclipse's JDT and platform bundles now
  come straight from Maven Central (they're published there in the release week of every
  Eclipse train). A clean build dropped from 48 s to **~10 s**; a train re-base is now an
  ordinary version bump. `mvn install` at the repo root is the whole story.
- **Boot: the Eclipse launcher + generated config are replaced by our own boot module**
  (~200 lines, the same pattern Eclipse's official Java language server uses in
  production): it assembles the bundle list, sets start levels, isolates sessions, and
  starts the framework programmatically. `java -jar jawata.jar -data <workspace>` works
  exactly as before.
- **Tests run in-framework, without Tycho.** The test suites are packaged as OSGi
  fragments of the bundles they test and executed by a small built-in JUnit launcher
  (`java -jar jawata.jar -runTests`). Full parity from the first run:
  **1183 tests, 0 failures** (1178 passed + 5 pre-existing skips). This also unlocks
  parallel local test runs (planned, Sprint 23) — impossible under Tycho.
- **One universal distribution.** Every platform's UI-toolkit fragment ships in the same
  archive and the runtime picks the right one at boot. The five per-platform download
  names remain (so installers and jawata-studio work unchanged) but now contain identical
  bits. macOS note: the layout is now the same `jawata-<version>/jawata.jar` directory as
  Linux/Windows — no more `Jawata.app` wrapper.

## Removed

- **The legacy `goja.*` names are gone, as announced.** The `goja.*` system-property
  fallbacks (`goja.experience.store`, `goja.workspace.root`, `goja.memory.roots`,
  `goja.experience.shared.dir`) and the `GOJA_*` environment-variable twins
  (`GOJA_TIMEOUT_SECONDS`, `GOJA_ABSOLUTE_PATHS`) are no longer read — promised in
  v2.7.1, delivered here. Use the `jawata` names. A value supplied only under a legacy
  name is now ignored (tested). Unaffected: the one-time store migration that rewrites
  goja-era data (package anchors, the old shared-store directory) still runs — old
  stores keep upgrading cleanly.
- **Dead cargo shipped since the first commit:** the m2e (Maven-integration), gson, and
  jdt.apt.core bundles were never referenced by any jawata code and are no longer
  bundled.

## Unchanged

- The **#5188 patch** (the locally repacked `compiler.batch` neutralizing
  [eclipse.jdt.core#5188](https://github.com/eclipse-jdt/eclipse.jdt.core/issues/5188))
  carries over into the new packaging as the only compiler.batch in the distribution;
  its sentinel test stays green and still gates removal when Eclipse 4.41 ships.
- Tool surface: 43 tools, names and behaviour identical (verified live against the
  v2.8.1 build: identical search results, byte-identical refactoring round-trips).

## Verification

- In-framework suite on the merged head: **1183 / 0 failures / 5 skipped** (baseline
  1179 + the 4 new legacy-name-ignored tests).
- Live parity vs the Tycho-built v2.8.1: 43 tool names diff-empty · compose_method
  plan→apply→undo byte-identical on both · identical search result sets on both.
