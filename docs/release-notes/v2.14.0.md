# jawata v2.14.0 — real engines under the refactoring tools

> **DRAFT (Sprint 25, Stage 3f).** The engine story below is implemented and
> gated; the `javadoc_lack` smell kind (Stage 4) and the RCP launch shape +
> `preset_args` (Stage 5) sections are added at their checkpoints before this
> ships.

**Sprint 25, phase 1.** Ten refactoring tools that used to build their output by
hand now run on the same engines Eclipse itself uses — the ones behind the IDE's
Refactor menu, shipped headless in `org.eclipse.jdt.core.manipulation`. Every
migration was gated the same way: we recorded the OLD tool's output on fixtures
first (goldens), migrated, and classified every difference — JDT resolving more,
formatting drift, or a new refusal — in a `DIVERGENCES.md` next to the golden.
Nothing was hidden; four of the replaced implementations had demonstrated
defects, and the migrations fixed real bugs.

**Still 45 tools.** Same names, same parameters (plus a few new optional ones).
Drop-in.

---

## What moved onto JDT engines

- **`rename_symbol`** — JDT's rename processors (type / method / field / local).
  Finds references the hand-rolled rewrite could miss.
- **`inline` (method)** — `InlineMethodRefactoring`. Drops the old tool's
  unconditional parentheses; otherwise byte-equal on the fixtures.
- **`inline` (variable)** — `InlineTempRefactoring`. Fixes a proven defect: the
  old tool destroyed sibling variables in `int a = 1, b = 2;`.
- **`change_method_signature`** — `ChangeSignatureProcessor` for the clean cases
  (rename, visibility, reorder/rename params, add-with-default, safe return
  changes): call sites update themselves, one undo. A COUPLED change (removing a
  used parameter, narrowing a return) is applied and answered with
  `coupledChange: true` plus the full worklist of introduced compile failures —
  the reported error locations ARE the edits to make. That break-then-fix
  workflow is deliberate and kept.
- **`extract` (variable / constant)** — `ExtractTempRefactoring` /
  `ExtractConstantRefactoring`. The old constant extraction produced a
  guaranteed syntax error on enum and record targets; gone.
- **`extract` (interface)** — `ExtractInterfaceProcessor`, with the
  workspace-wide "use the interface where possible" rewrite: compatible
  declarations (locals, even a factory's return type) are retyped to the new
  interface. The default selection no longer extracts supertype overrides
  (`toString`, `compareTo`) — extracting them produced a broken interface, which
  the old tool masked by copying signatures verbatim.
- **`extract` (superclass)** — `ExtractSupertypeProcessor` is the new DEFAULT:
  fields, constructors, generics, members the siblings don't share, and
  duplicate deletion in the siblings. The old conservative contract (only
  byte-identical, self-contained methods across all named types) is preserved
  unchanged under **`mode=identical`** — it pairs with `copy_class` where the
  byte-identical guarantee is the point.
- **`organize_imports`** — `OrganizeImportsOperation`, the real thing: ADDS
  missing imports, prunes unused static imports, honors project-configured
  import order. The old tool could only remove and sort — its own golden proved
  it never pruned a static import. The workspace sweep shell (honest missed-file
  reporting) stays ours.
- **`convert_anonymous_to_lambda`** — `LambdaExpressionsFixCore`. The old tool
  silently ignored fields declared in the anonymous class; the engine refuses
  what it cannot convert and removes imports the conversion orphans.
- **`apply_cleanup`** — JDT's fix cores (`VariableDeclarationFixCore`,
  `RedundantModifiersCleanUp`) inside our sweep shell. Catches the cases the
  hand-rolled scans missed (catch/enhanced-for parameters, nested enums and
  records).

## What stayed hand-rolled — and why, honestly

**No engine exists** (verified against the shipped jars, not assumed):

- The pattern transforms — `inline_singleton`, `refactor_to_state`,
  `refactor_to_command_dispatcher`, `refactor_to_visitor`,
  `form_template_method`, `replace_type_code_with_class`. Kerievsky-style
  pattern moves; JDT has nothing in this space.
- `replace_duplicates` — clone-group delegation to a canonical method.
- `generate(copy_class)` — clone a class into a sibling file (pairs with the
  conservative superclass mode).

**An engine exists, ours kept by decision** (each with the recorded reason):

- `change_method_signature`'s **retarget mode** — JDT's
  `ReplaceInvocationsRefactoring` inlines the target's body into callers rather
  than retargeting the call, and its own source admits unsupported cases
  (constructor invocations: "not yet"). Ours retargets by name and handles
  constructors. Contributing that upstream is on the roadmap.
- **`apply_null_annotations`** — JDT's `NullAnnotationsFixCore` is
  problem-driven (it fixes existing null-analysis diagnostics); ours is
  target-driven (annotate THIS symbol). Different problem; nothing to migrate
  onto.
- **The `generate` family** — JDT's `GetterSetterUtil` is classic-style only;
  our AST builder is the more capable base. Instead of migrating we closed its
  gaps: **record-style getters** (`x()`), **fluent setters** (return `this` for
  chaining), **Javadoc emission**, and **prefix-aware naming** (a `fName` field
  now yields `getName`, via JDT's public `NamingConventions`).
- **`extract(kind=superclass, mode=identical)`** — the conservative
  byte-identical contract, kept as described above.

## New since v2.13.1 (beyond the engines)

- `generate(kind=getters_setters)`: `getterStyle` (`classic`|`record`),
  `setterStyle` (`classic`|`fluent`), `generateJavadoc`.
- `extract(kind=superclass)`: `mode` (`jdt` default | `identical`), and
  `members` now accepts fields in jdt mode.
