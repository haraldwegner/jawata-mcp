# GOJA MCP v2.2.5 — memory-file names become recall cues

Point release on v2.2.4.

## Fix: the loader indexes the frontmatter `name` as a symptom

A memory file's **name** is where the cue-dense phrasing lives — *"Tauri webview renders
**blank** on aarch64"* — while its description may use different words ("content area
stays the GTK background colour"). The loader used to drop the name entirely, so a
natural cue like `blank webview` couldn't reach an entry that plainly answers it.

Names are now indexed as symptoms at load (alias-normalized, searched by both recall
phases like any recorded symptom). No schema change; re-run `load` once after updating —
changed entries re-ingest and pick up their name cues.

The remaining gap — cues whose words appear in *neither* name nor description ("blank"
vs "background colour" as true synonyms) — is semantic matching, deliberately reserved
for the embeddings track (Sprint 27) behind the same fit gate.
